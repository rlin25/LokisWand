# Walkthrough: loki_candidate_profile.md

**Source file:** `profile/loki_candidate_profile.md`
**Role:** The primary LLM input — the stable, structured context that both assessment instances receive for every job evaluation

---

## Purpose

The candidate profile is not documentation. It is an input to a machine. Every structural choice in it — the Review Required section placement, the flat skills lists, the technical details field, the highlight tags — exists to improve the reliability of the Claude assessments that consume it. Understanding why the document is structured this way requires understanding what breaks when it is structured differently.

---

## Relationships

**Generated by:** `profile/loki_conversion_prompt.md` — a one-time Claude conversation operation

**Consumed by:** `workflows/loki_core_pipeline.json` — embedded in the Load Profile Document code node, passed as shared context to Instance A and Instance B HTTP Request nodes

**Not referenced by:** The follow-up nudge workflow, which only reads Airtable fields and has no knowledge of the profile

---

## Decision rationale

### Review Required section appears first, unconditionally
Hard constraints, goals, and known gaps always appear in the Review Required section regardless of Claude's confidence level during generation (Decision 5). This is unconditional because these are the fields most likely to be wrong when inferred and most costly when wrong. A location constraint that Claude inferred incorrectly would cause Instance B to miss a genuine disqualifying risk signal on every application until corrected.

Placing Review Required first means the user cannot miss it during the one-time review step — it is the first thing visible, not buried after a long profile body.

### Flat skills lists without proficiency levels
Skills are listed by category with no proficiency ratings attached (Decision 7). The reasoning: self-reported proficiency levels degrade in accuracy over time and introduce a maintenance burden that users rarely keep current. More importantly, an inaccurate proficiency tag produces a confidently wrong assessment. Claude infers relative skill strength from the experience and highlights sections, which carry more honest signal — a skill used to accomplish something meaningful in a specific role says more than a "proficient" label applied at resume-writing time.

### Technical details field alongside achievement bullets in each experience entry
Each experience entry has two distinct sections: Achievements (impact-focused, written for human readers) and Technical details (stack-focused, written for LLM assessment) (Decision 8). This separation exists because achievement language — "reduced processing time by 40%" — provides no information about what was built or what stack was used. Instance A cannot map a vague achievement to a specific technical requirement without inferring the technology from context, which introduces variance. The technical details field gives both instances the concrete evidence they need without requiring inference.

Note: the PrismForge AI and OwnerFi technical detail fields contain explicit verification notes ("Verify") because the resume did not state specific framework names for those roles. This is the correct behavior from the conversion prompt — flagging uncertainty rather than guessing.

### Explicit relevance tags on each highlight
Each highlight in the Highlights and Accolades section carries short, specific tags (Decision 9). Highlights are the highest-signal assets in the profile: they are the candidate's best projects and most distinctive achievements. Without tags, Instance A must infer from the description which highlights are relevant to the specific role — and that inference varies between calls. Tags give Instance A a deterministic path to the most relevant highlights rather than requiring role-to-highlight relevance to be re-derived on every submission.

### Version field (v1.0)
The version field in the header is logged into every Airtable record at assessment time (Decision 15). Its presence in the profile document is what makes that logging possible — the Load Profile Document code node reads the version field from the embedded profile text and writes it to the Airtable record. Without the version field, assessments cannot be audited after a profile update.

---

## What it does not do

- **Does not contain proficiency levels** — by design (Decision 7)
- **Does not embed job-specific context** — the profile is stable across all assessments; job context is always the pasted job description injected separately
- **Does not self-update** — any change to the profile requires a manual file edit and a workflow update to re-embed the new version
- **Does not hide the Review Required section from assessment instances** — both Instance A and Instance B receive the full profile including the Review Required section. This is intentional: hard constraints in Review Required (like location constraints) are relevant evidence for risk signal identification

## v2 touch points

- **PrismForge AI and OwnerFi technical detail gaps**: Both entries note that specific frameworks and tooling were not stated in the resume. These should be filled in explicitly in the next profile update (profile version v1.1 or v2.0).
- **Known Gaps currency**: The four known gaps identified (ML model training at scale, data engineering pipelines, cloud certifications, team leadership) will change as skills develop. Profile updates that close a known gap should remove it from the Known Gaps section and increment the version field — so any Airtable records using the old profile version can be identified and reconsidered.
- **External profile storage**: The profile is embedded in the workflow rather than loaded from a file. When the profile is updated, the workflow must be edited. v2 should externalize the profile to an n8n variable or static file reference to break this coupling.
